**Document structure — include ALL of these sections:**

### Section 1: Context
- This is a Claude Code prompt for the main Tenseats marketplace repo
- Purpose: implement the complete post-signup experience (verification, success pages, onboarding persistence)
- Relationship to prelaunch site: same Convex backend, separate Next.js apps
- Instruction: "Always check the current schema before adding new tables or fields"

### Section 2: Schema Approach
- Prefer existing fields over new tables (per user decision)
- Map existing fields to onboarding steps:
  - Role selection: `userRoles` table (always complete by signup)
  - City selection: `userPreferences.currentMetroId`
  - Cuisine preferences: `userPreferences.cuisinePreferences`
  - Stripe connect (vendors only): `users.stripeCustomerId`
  - Overall completion: `users.isProfileComplete`
- If a new `onboardingProgress` table is truly needed after reviewing current schema, keep it minimal
- Include schema verification instructions: "After any schema changes, run `npx convex deploy` and verify the new fields are queryable"

### Section 3: Redirect Flow
- Specify the complete chain: signup -> "almost there" page -> OTP email verification -> role-based success page
- Role routing logic: query `userRoles` table for the authenticated user
  - Role is `guest` -> guest success page
  - Role is `chef`, `mixologist`, `creator`, or `venueHost` -> vendor success page
- Note: 5 roles total (chef, mixologist, creator, venueHost, guest). No "facilitator" in schema.
- Address the verify-email session guard tech debt: the prompt must ensure unauthenticated users arriving at verify-email are handled gracefully (redirect to signup or show appropriate message)

### Section 4: Pages to Build
For each page, specify: exact route path, component file path, component name, props/data requirements, and exact consumer-facing copy.

**Page 4a: "Almost There" verification page**
- Route: specify appropriate route path
- Shared page for both guest and vendor signups (not role-specific)
- Content: branded "check your email" messaging
- Use the tenseats-copy-writer skill (read brand-voice.md and accepted-copy.md first) to write exact headline, subheadline, body copy, and any CTA text. The copy should be warm and distinctive — not generic "please verify your email" text
- Design guidance: distinctive, warm design — not a generic form confirmation screen

**Page 4b: Guest success page**
- Route: specify appropriate route path
- Shown after OTP verification for guest-role users
- Content sections:
  1. Welcome-to-community inspirational messaging (exact copy via copy-writer skill)
  2. Address book reminder: "Add supportteams@tenseats.io to your address book" with clear rationale
  3. Social links: Instagram (https://www.instagram.com/tenseats) and Pinterest (placeholder URL) with custom icon components
- Next onboarding steps: city selection, cuisine preferences (link to relevant settings pages)

**Page 4c: Vendor success page**
- Route: specify appropriate route path
- Shown after OTP verification for vendor-role users (chef, mixologist, creator, venueHost)
- Content sections:
  1. Welcome messaging appropriate to vendor role (exact copy via copy-writer skill)
  2. Optional Stripe connect step: "Connect your Stripe account" — clear messaging that this can be done now or at launch. Use Stripe Express Connect flow. Link/button to initiate Stripe OAuth
  3. Zoho One recommendation: "one fee for 50+ best-in-class tools" with link to https://go.zoho.com/Slvq — this is informational only, no technical integration
  4. Social links: same Instagram and Pinterest links as guest page
  5. Address book reminder: same as guest page

### Section 5: Onboarding Persistence Logic
- How to check completion: query existing fields, not a separate progress tracker
- Vendor-specific: check `users.stripeCustomerId` for Stripe step
- Common: check `userPreferences.currentMetroId` for city, `userPreferences.cuisinePreferences` for cuisine
- `users.isProfileComplete` flag: set to true when all required steps are complete (define what "required" means per role)
- When user returns after logging out, success page should show which steps are already complete and not re-prompt

### Section 6: Social Links & Icons
- Instagram: https://www.instagram.com/tenseats
- Pinterest: placeholder URL (note: professional page URL TBD, use placeholder)
- Icons: custom monoline SVG components matching brand aesthetic (note: prelaunch site builds its own in Phase 21, main app needs its own)

### Section 7: Design Requirements
- All three pages (almost-there, guest success, vendor success) must share a coherent design language
- Distinctive, high-quality frontend design — not generic AI aesthetics
- Reference the instagram-ui skill if available in the main app repo, otherwise specify: black/white color system, Tailwind + shadcn/ui, WCAG AA, mobile-first

### Section 8: Gotchas & Edge Cases
- verify-email session guard tech debt
- Role must be written to `userRoles` during signup flow if not already
- Handle case where user has no role yet (edge case — redirect to role selection)
- Stripe connect is optional — vendor success page must work without it
- Copy must be verbatim as specified (use copy-writer skill output)

**Copy writing process:** For every consumer-facing text element in the prompt (headlines, subheadlines, body copy, CTAs, button labels), use the tenseats-copy-writer skill workflow:

3. Generate copy that is warm, evocative, direct — never corporate or generic
4. Include the exact copy in the prompt document so the main app implements verbatim
